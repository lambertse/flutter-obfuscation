# Flutter Android AOT-snapshot packer

A build-time + runtime pair that encrypts `libapp.so`'s Dart AOT snapshot at
rest and decrypts it in memory, transparently, at app startup. Implements
`Handover.md`. **Read `Handover.md` first** -- this README covers what was
built and how to run it; the handover has the full design rationale and
acceptance criteria this implementation is judged against.

## What this does and does NOT protect (plain English)

**Defeats:** a casual attacker who unzips the APK and runs `strings` / a
decompiler / a Dart snapshot parser directly against `lib/<abi>/libapp.so`
on disk. That's it.

**Does NOT defeat:**
- **A live memory dump** (Frida, ptrace, `/proc/<pid>/mem`). The snapshot
  is plaintext in executable memory the whole time the app is running --
  encryption-at-rest raises the cost of *static* analysis, nothing more.
  `native/libguard/src/anti_instr.c` is a **v1 no-op stub**; see "v2 plan"
  below for what would actually narrow this window.
- **Symbol obfuscation.** This packer is worthless without it --
  `build_and_pack.sh` refuses to run if `flutter build --obfuscate` doesn't
  look like it actually obfuscated the app's own Dart libraries (step 3,
  the "obfuscation gate").
- **A determined reverse engineer with time.** This raises cost, it does
  not create a hard boundary. Communicate it internally as such.

## v1 scope decision: instructions-only

`Handover.md` §13 asked instructions-only vs. instructions+data; this build
does **instructions-only** (the two executable `.text` regions:
`_kDartVmSnapshotInstructions`, `_kDartIsolateSnapshotInstructions`). That
means:

- The two `.rodata` **data** regions (`_kDartVmSnapshotData`,
  `_kDartIsolateSnapshotData`, ~1.8-3.6MB combined) stay **plaintext**,
  including residual `package:`/class-name strings. Confirmed present via
  static analysis of the reference test APK (438 `package:` string
  occurrences, including the app's own paths, found in the unencrypted
  `.rodata` region).
- **This means Handover.md §10's acceptance criterion "`strings
  lib/<abi>/libapp.so | grep package:` on the shipped file returns ~nothing"
  will NOT pass as literally stated** under this v1 config -- only the
  instruction regions are ciphertext. If that acceptance criterion needs to
  hold literally, flip `ENCRYPT_DATA_REGIONS = True` in
  `tools/encrypt_snapshot.py` (one line; the native side needs zero changes,
  `trampoline.c` just iterates whatever `regions.h` contains).

## Repo layout

```
packer/
  native/libguard/          # the runtime .so (C, built via CMake + Android NDK)
    CMakeLists.txt
    src/
      guard.c                # constructor + JNI key hand-off
      got_hook.c/.h           # PLT/GOT patch of dlopen in libflutter.so
      trampoline.c/.h         # the installed dlopen hook + decrypt_region()
      crypto.c/.h             # AES-256-CTR wrapper over vendored tiny-AES-c
      region_table.h          # guard_region_t struct (shared contract)
      regions.h               # GENERATED by encrypt_snapshot.py -- not checked in
      anti_instr.c/.h         # v1 stub, see "v2 plan" below
      log.h
      third_party/aes.c/.h    # vendored tiny-AES-c (Unlicense), see PROVENANCE.md
  tools/
    encrypt_snapshot.py       # build-time: readelf + AES-CTR encrypt + emit regions.h
    build_and_pack.sh         # end-to-end pipeline, WITH Flutter source (see below)
    pack_existing_apk.sh      # end-to-end pipeline, APK ONLY / no source (see below)
    axml_patch.py             # surgical binary AndroidManifest.xml attribute patcher
    requirements.txt
  android/                    # Path B (you have Flutter source): Kotlin, Gradle-compiled
    GuardBridge.kt             # Kotlin: PackageManager cert read + HKDF key derivation
    App.kt                     # TODO: adjust package, wire into AndroidManifest.xml
    gradle-notes.md            # packagingOptions / manifest notes
  apk-inject/                 # Path A (APK only, no source): standalone Java module,
    src/dev/packer/guard/      # compiled independently and injected as classesN.dex --
      GuardBridge.java          # NOT part of the target app's own build. Same logic as
      GuardApplication.java     # android/GuardBridge.kt+App.kt, ported to plain Java.
    build.sh                   # compiles + dexes the above via javac + d8
  README.md                    # this file
```

Two parallel pipelines, both producing the same runtime result
(`libguard.so` decrypting `libapp.so` at startup):
- **`pack_existing_apk.sh`** (Path A, `docs/GUIDE.md`): takes an
  already-built `.apk`, no source needed anywhere. Patches the compiled
  `AndroidManifest.xml` (`axml_patch.py`) to point `<application
  android:name=...>` at a precompiled `GuardApplication`
  (`apk-inject/`, injected as a new `classesN.dex`), then does the same
  encrypt/build-libguard/repack/sign steps as build_and_pack.sh.
- **`build_and_pack.sh`** (Path B): you have the Flutter project source;
  add `android/GuardBridge.kt`+`App.kt` to it and let Gradle compile them
  normally as part of a real `flutter build --obfuscate`.

## Running it

Full step-by-step for both paths (including Path A / no-source, the
validated one): [`docs/GUIDE.md`](../docs/GUIDE.md). Quick reference for
Path B (you have Flutter source):

```
pip install -r packer/tools/requirements.txt

export ANDROID_NDK_HOME=/path/to/ndk
export ANDROID_BUILD_TOOLS=$ANDROID_SDK_ROOT/build-tools/<version>

# release build:
export KS_PASS=...    KEY_PASS=...
packer/tools/build_and_pack.sh \
  --project-dir /path/to/flutter/app \
  --keystore /path/to/release.jks --key-alias upload \
  --keystore-pass-env KS_PASS --key-pass-env KEY_PASS

# dev/local build (snapshot key is NOT derived from a real release cert --
# do not ship this):
packer/tools/build_and_pack.sh \
  --project-dir /path/to/flutter/app \
  --keystore /path/to/debug.jks --key-alias debugkey --keystore-pass-env KS_PASS \
  --dev-key-hex $(openssl rand -hex 32)
```

Output: `build/packed/app-release-packed.apk`, signed and zipaligned.
Debug symbols for `flutter symbolize` land in `build/packed/debug-info/`.

Before wiring `App.kt`/`GuardBridge.kt` into your actual app: rename the
package in `App.kt` (top of file) and register it as
`android:name=".App"` in `AndroidManifest.xml` -- see
`android/gradle-notes.md`.

## Why key derivation is in Kotlin, not C

`Handover.md` §8 specifies `HKDF(sha256(cert_der) || constant)`. Reading the
release signing cert needs a live `Context`/`PackageManager`, which is only
reachable from Java/Kotlin (or hidden-API reflection tricks from native
code, deliberately avoided -- see `GuardBridge.kt`'s doc comment). Since
that JNI round-trip is required anyway, `GuardBridge.kt` also does the
SHA-256/HMAC/HKDF hashing there via `java.security.MessageDigest` /
`javax.crypto.Mac` (JCA, audited) rather than vendoring a *second* native
crypto primitive on top of AES. `guard.c`'s native side only ever sees the
already-derived 32-byte key.

**This cross-language contract has been verified.** A direct Java port of
`GuardBridge.kt`'s `hkdfSha256`/`hmacSha256`/`sha256` (same JCA calls:
`MessageDigest.getInstance("SHA-256")`, `Mac.getInstance("HmacSHA256")` --
Kotlin and Java compile to identical bytecode for these calls, so this is
not a simulation) was run against a portable JDK and compared to
`encrypt_snapshot.py`'s `derive_key_from_cert()` (Python, `cryptography`
package) across four test inputs (4, 500, 1200, and 1-byte "certificates").
**All four produced byte-identical 32-byte keys.** This does not verify the
Kotlin *syntax* compiles (no Kotlin compiler was available, only `javac`)
or the `PackageManager`/`SigningInfo` cert-reading path (needs a real
Android runtime) -- but the actual cryptographic contract both sides must
agree on, byte-for-byte, is confirmed correct.

## AES-CTR byte contract

`crypto.c` / `encrypt_snapshot.py` must agree exactly:
AES-256-CTR, `IV = 12-byte nonce || 0x00000000` (4-byte big-endian counter,
starting at 0), full 128-bit big-endian counter increment per block. A
fresh random nonce is generated per `(ABI, region)` on **every build run**
(not a fixed formula) specifically so the same stable, cert-derived key
never reuses a keystream across two different app builds. **This exact
construction was verified against real bytes extracted from the reference
test APK** (round-trip correctness, untouched-byte-range integrity, and the
128-bit counter-increment semantics were all checked with Node's `crypto`
module and cross-referenced against tiny-AES-c's source before writing
`crypto.c`/`encrypt_snapshot.py` -- this is the one piece of the pipeline
that had an empirical check available and it was used).

## What has actually been run, not just reviewed

The authoring sandbox started with none of the required tooling (no
Android NDK/SDK, no C compiler, no `python3`, no `readelf`, no JVM,
no `zip`/`zipalign`/`unzip`, no network-independent x86_64 emulation --
this is an aarch64 Linux host, and Android's own NDK/build-tools only ship
x86_64-host binaries). Rather than ship everything as reviewed-but-untested,
a full toolchain was assembled from portable, no-root downloads
(Temurin JDK, a real Android NDK's sysroot + `android.toolchain.cmake`,
Zig 0.16 as a natively-runnable aarch64-host clang+lld frontend paired with
that real NDK sysroot via `zig cc --libc`, a portable Python +
`cryptography`, and hand-written Node.js reimplementations of zip-update
and `zipalign` where the real CLI tools couldn't run natively) and the
**entire pipeline was actually executed** against the reference test APK:

- **`encrypt_snapshot.py` was actually run** against all 3 ABIs' real
  `libapp.so` files extracted from `assets/app-release.zip`. Output
  addresses/sizes matched the values independently computed by hand during
  the initial ELF analysis, exactly. AES-CTR round-trip decryption was
  verified against the real encrypted output, and all bytes outside the
  target regions were confirmed byte-identical to the original.
  (`readelf` still isn't available in this sandbox -- `encrypt_snapshot.py`
  now has an automatic pure-Python ELF32/ELF64 fallback parser,
  `_MiniElf`, used whenever `readelf` isn't on PATH. This is what was
  actually exercised; get `readelf` on your CI and run it once to confirm
  the primary path too.)
- **`libguard.so` was actually compiled and linked for all 3 ABIs**
  (arm64-v8a, armeabi-v7a, x86_64) against the real Android NDK r28c
  sysroot, producing valid ELF (AArch64 ET_DYN, ARM/ELF32 ET_DYN, x86-64
  ET_DYN respectively), each with `Java_dev_packer_guard_GuardBridge_
  nativeSetKey` correctly present in `.dynsym` (GLOBAL, defined) and
  everything else hidden per `-fvisibility=hidden`, and `.init_array`
  present (confirming the constructor is registered). **This caught a real
  bug**, now fixed: `CMakeLists.txt` applied `-DAES256=1` (and the other
  tiny-AES-c mode defines) only to `third_party/aes.c`'s compile command
  via `set_source_files_properties`, but `crypto.c` also `#include`s
  `third_party/aes.h` directly and has a `_Static_assert` checking
  `AES_KEYLEN == GUARD_AES_KEY_LEN` -- without the same define,
  `crypto.c`'s translation unit saw the header's AES-128 default and
  failed to build. Fixed by moving the defines to
  `target_compile_definitions` (target-wide) instead.
- **The full repack pipeline was actually run**: the real `libapp.so`
  (encrypted) and `libguard.so` (compiled above) were injected into a copy
  of the reference APK, page-aligned to 4096 bytes for all STORED `.so`
  entries, and signed with the real `apksigner.jar` (via the portable
  JDK) using a throwaway keystore. **`apksigner verify` reports
  `Verifies`, with both the v2 and v3 signature schemes validated true.**
  Every untouched entry (`resources.arsc`, `classes.dex`,
  `AndroidManifest.xml`, ...) was confirmed byte-identical to the original
  APK before and after repacking, and alignment was re-verified to hold
  after signing.
- **The HKDF cross-language contract was verified directly** -- see "Why
  key derivation is in Kotlin, not C" above.
- **The no-source (`pack_existing_apk.sh` / Path A) pipeline was built and
  run for real, end to end**, including its two novel, previously-unbuilt
  pieces:
  - `apk-inject/build.sh` actually compiled `GuardBridge.java` +
    `GuardApplication.java` (against a real downloaded `android.jar`) and
    `d8`-dexed them into a `classesN.dex`, confirmed (by parsing the dex's
    own `class_defs`) to contain exactly the 2 real classes.
  - `axml_patch.py`'s binary manifest edit was run against the real
    `AndroidManifest.xml` from the reference APK, and independently
    re-verified by dumping *every* element/attribute before and after: of
    58 elements, exactly one attribute value differs
    (`<application android:name=...>`), everything else -- including
    every other attribute on that same element -- byte-for-byte identical.
    Two real bugs were caught this way (both fixed): the element-name
    field offset was initially read from the wrong byte position (found
    every element's name resolving to garbage), and the attribute-array
    base offset was computed from an assumed-but-wrong interpretation of
    the `attrStart` field (fixed by using the empirically-confirmed fixed
    36-byte header layout instead, matching how AOSP's own reader and
    community tools like androguard actually walk this structure).
  - The full result -- patched manifest + injected dex + encrypted
    `libapp.so` + compiled `libguard.so`, all in one repack -- was aligned
    and signed with the real `apksigner.jar`, and **`apksigner verify`
    passed** (v3 signature). Re-extracting the signed output confirmed the
    manifest still reads the patched value, the injected dex is
    byte-identical to what was built, and `classes.dex`/`resources.arsc`
    are untouched.
- **`pack_existing_apk.sh` was then run end-to-end on a real machine (real
  NDK, real `apksigner`, not this sandbox's substitutes) against a real
  APK, and installed on a real device -- and it crashed on launch.** This
  is the actual device-testing feedback loop the "Known limitations"
  section below kept flagging as unprovable in-sandbox, and it caught a
  real bug the sandbox categorically could not have caught:
  `GuardApplication.onCreate()` originally called
  `new FlutterLoader().startInitialization(this)` (mirroring
  `App.kt`/Handover.md's own suggested approach) to load `libflutter.so`
  before installing the GOT hook. On the real device,
  `io.flutter.embedding.engine.loader.FlutterLoader` turned out to have
  been fully removed from that build's `classes.dex` by R8 (confirmed by
  grepping the extracted dex for class descriptors: every other
  `io/flutter/embedding/engine/*` class had one, `.../loader/FlutterLoader`
  had none -- only its name survived inside unrelated log-message string
  literals), causing an immediate `NoClassDefFoundError` in
  `Application.onCreate()`. **Fixed** by dropping the `FlutterLoader`
  dependency entirely: `GuardApplication`/`App.kt` now call
  `System.loadLibrary("flutter")` directly. This is not just a fix for one
  broken build -- it's strictly more robust in general, since the GOT hook
  only ever needed `libflutter.so` mapped into the process (a native-level
  fact), never anything from Flutter's Java API surface, which is exactly
  the part R8/Flutter-version changes can rename or remove out from under
  you. The `apk-inject/compile-only-stubs/` scaffolding this used to need
  (to satisfy `javac` without a real Flutter jar) is gone along with the
  dependency.

Concretely, `build_and_pack.sh` itself (the literal Bash script, invoking
the literal `flutter`/`cmake`/`zip`/`zipalign`/`apksigner` CLIs) was **not**
run end-to-end, for two reasons that are inherent to this environment, not
fixable by installing more tools: (1) there is no Flutter project *source*
available here, only the already-built test APK, so the
`flutter build --obfuscate` step itself cannot be exercised; (2) there is
no Android device or emulator available, so runtime behavior (does the GOT
hook actually fire correctly, does `mprotect` succeed on a real device, does
the app actually launch) is unverified. Every stage that *could* be
exercised without those two things was, using substitute tools chosen to
preserve the actual logic being tested (e.g. `zig cc --libc <ndk-sysroot>`
compiles against the *real* NDK sysroot and produces *real* Android ELF
binaries -- it is not a simulation of compilation, just a different
compiler frontend than the NDK's own x86_64-only `clang` binary, which
cannot run on this host at all).

Remaining known gaps, now much narrower given the above:

1. **W^X / SELinux `execmod` fallback: implemented and CONFIRMED WORKING
   on a real device.** This started as the theoretical gap `Handover.md` §9.1 warns
   about, then was empirically confirmed: a real OnePlus 6T (Android 11,
   enforcing SELinux, non-rooted -- an unremarkable stock device) hit
   `avc: denied { execmod }` on `base.apk` the very first time the packer
   ran, meaning the naive in-place `mprotect(RW)`\-\>decrypt\-\>`mprotect(RX)`
   sequence fails on essentially every modern device, not just a corner
   case. A same-address per-region `mremap`/remap trick was considered and
   rejected early on for the reason still true today: the two instruction
   regions in the reference APK sit **32 bytes apart on the same page**, so
   naive page-granular remapping of one region would clobber the other.
   The doc's other suggested fallback (feed decrypted pointers to the
   engine via its native embedder API, bypassing `dlopen` entirely) was
   ruled out as far more invasive -- it requires replacing the standard
   Java `FlutterActivity`/engine embedding with a custom native entry
   point.

   **Implemented instead:** `trampoline.c`'s `decrypt_exec_regions()`
   computes the page-aligned span covering *all* exec regions together (not
   per-region), decrypts into a scratch buffer first, then atomically swaps
   the live file-backed mapping for anonymous memory at the *same* virtual
   address (`mmap(..., MAP_FIXED | MAP_ANONYMOUS)`) before marking it
   `PROT_EXEC`. Anonymous memory was never file-backed, so SELinux's
   `execmod` class does not apply to it -- only `execmem` does, which is a
   normal, always-granted permission for `untrusted_app` (this is exactly
   how JIT engines get executable pages on Android). Keeping the same
   address avoids the relocation risk of moving the instructions elsewhere:
   no `dlsym` hook or redirect table is needed, and no assumption about
   Dart AOT position-independence is required. Non-exec (data) regions are
   unaffected by any of this and still use the simple in-place path
   (`execmod` is specifically about the *executable* transition).

   Verified on the OnePlus 6T: the device log shows both instruction
   regions decrypted, `replaced 3141632-byte span ... with anon RX mapping`,
   and **no `execmod` denial** -- the app now progresses past this point.
   (The `MAP_FIXED`-over-linker-mapping concern raised during design turned
   out to be a non-issue in practice, as expected since Flutter never
   `dlclose()`s `libapp.so`.)

2. **Snapshot decrypts to garbage after the `execmod` fix -- suspected
   build-vs-runtime key mismatch, under active diagnosis.** With `execmod`
   solved, the next device run got further and then `SIGSEGV`'d inside
   `libflutter.so` during `FlutterJNI.performNativeAttach`, dereferencing a
   wild pointer -- the signature of the engine reading a snapshot whose
   *instruction* bytes are wrong. Ruled out in the sandbox, no device
   needed: (a) the AES-CTR primitive round-trips byte-for-byte across the
   Python-encrypt / C-decrypt language boundary at the exact region sizes
   (92768, 3046944); (b) the packer's symbol->file-offset conversion
   (`vaddr_to_file_offset`, section-header based) produces the *same* offset
   as the program-header (`PT_LOAD`) based conversion that `dlsym` implies
   at runtime, and both regions lie fully inside the R+X segment -- so the
   packer encrypts exactly the bytes `dlsym` resolves; (c) the device log
   confirms `decrypt_exec_regions` hit the correct addresses/sizes/span.
   That leaves the AES **key**: the build derives it from the cert exported
   by `keytool -exportcert`, the runtime derives it from
   `apkContentsSigners()`; if those cert DER bytes differ, the key differs
   and the instructions decrypt to garbage. To confirm without guessing,
   `encrypt_snapshot.py` and `GuardBridge` now log matching non-secret
   fingerprints (`cert-fp`, `key-fp` = first 4 bytes of each SHA-256, tag
   `libguard`). Next device run: compare the pack-time line with the
   `logcat` line -- `cert-fp` differs => different signing cert; `cert-fp`
   matches but `key-fp` differs => Python/Java HKDF disagree; both match =>
   the diagnosis is wrong and the mechanism needs another look.

3. **`anti_instr.c` is a v1 no-op stub**, as explicitly permitted by
   `Handover.md` §12 deliverable 5. This is the layer that would actually
   defend the decrypted-in-memory window (ptrace/Frida/debugger-attach
   detection) -- encryption-at-rest alone does not, see "what this does
   NOT protect" above. **v2 plan:** `ptrace(PTRACE_TRACEME)` self-attach
   check, `/proc/self/status` `TracerPid` polling, and a
   `/proc/self/maps` scan for known Frida gadget/agent library names,
   invoked from `guard_ctor()` and periodically from a background thread.
   Needs real-device testing against Frida specifically before being
   trusted -- do not ship anti-Frida logic that hasn't been tested against
   Frida.

4. **The GOT-hook's `DT_SYMTAB`/`DT_STRTAB`/`DT_JMPREL` address computation
   assumes `base + d_ptr`** (the standard convention used by essentially
   every open-source Android PLT-hook implementation: xHook, legend, etc.).
   A small number of toolchains/loaders are known to pre-relocate these
   specific `.dynamic` pointer fields, which would make this double-add the
   load bias. `got_hook.c` defends against acting on a wrong address by
   bounds-checking every computed pointer against the target module's own
   `PT_LOAD` ranges before dereferencing it (fails closed -- logs and
   returns -1, never crashes) -- but this has not been exercised against a
   device where the assumption is actually wrong. Confirm on the device
   test matrix.

5. **`readelf` text-output parsing in `encrypt_snapshot.py` still hasn't
   exercised the real `readelf` binary** -- no `readelf` was available in
   the authoring sandbox, so what actually ran (and was verified end-to-end
   against the real reference APK, all 3 ABIs) was the `_MiniElf` fallback
   path, not the primary `readelf`-parsing path. The two paths return the
   same shape of data (`{symbol: (addr, size)}`, `[(name, addr, offset,
   size)]`) and `_MiniElf` was built to match `readelf`'s own field
   semantics exactly, but the `readelf` text-regex path
   (`_SYM_RE`/`_SEC_RE`) itself was only checked against hand-constructed
   sample lines via a Node.js proxy (including the empty-name `NULL`
   section edge case, which the naive regex mis-parses -- fixed), not
   against real `readelf` output. Run it once with real `readelf` on PATH
   and diff the resulting `regions.h` against a `_MiniElf`-produced one for
   the same input before trusting the `readelf` path in CI.

6. **Startup ordering is load-bearing, not incidental.** `guard_ctor()`
   patches a GOT slot *inside libflutter.so*, which means libflutter.so
   must already be mapped when `System.loadLibrary("guard")` runs, or the
   hook install fails and the process aborts (fail-closed by design). A
   first draft of `App.kt` referenced `GuardBridge` from a companion
   `init` block to force early class-load-time loading -- that does NOT
   work, because nothing loads libflutter.so that early on a plain
   `Application`; it's loaded lazily later by the engine's own init path.
   The fix (caught during review): `App.kt.onCreate()` explicitly loads
   libflutter.so *before* `GuardBridge.install(this)`. This ordering has
   now been confirmed correct on a real device -- see the
   `System.loadLibrary("flutter")` bug entry above for the one part of
   this that review couldn't catch (*how* to load it, not *when*).

7. **`build_and_pack.sh`'s repack step updates the original APK in place**
   (copy + targeted `zip -0` replace/add of just `lib/<abi>/libapp.so` and
   `lib/<abi>/libguard.so`) rather than rebuilding the archive from
   scratch -- caught and fixed during review (a first draft rebuilt the
   whole zip with its own compression policy, which would have
   recompressed `resources.arsc` along the way; Android 30+ requires that
   file to stay STORED and aligned, so that draft would have produced an
   APK that fails to install with `INSTALL_PARSE_FAILED_*` on any such
   device). **The corrected approach was then actually exercised** end to
   end (see "What has actually been run" above): every untouched entry
   confirmed byte-identical, all STORED `.so` entries confirmed 4096-byte
   aligned both before and after signing, and `apksigner verify` confirmed
   the result installs a valid v2/v3 signature. What's still unconfirmed is
   the literal `zip`/`zipalign` CLI invocations in `build_and_pack.sh`
   itself -- the sandbox has neither binary, so this was validated via
   hand-written Node.js reimplementations of the same operations
   (replace-in-place with CRC32 recompute; 4096-byte page alignment via
   local-header extra-field padding) rather than the real tools. The real
   `zip`/`zipalign` are simpler, more mature, and should behave at least as
   correctly -- but run the actual script once on a machine that has them
   before shipping.

8. **JNI class path is hardcoded** (`dev.packer.guard.GuardBridge`, chosen
   deliberately so it doesn't need to match your app's actual package --
   see `GuardBridge.kt`/`GuardBridge.java`, which share this constraint).
   If you move that file, `guard.c`'s
   `Java_dev_packer_guard_GuardBridge_nativeSetKey` export name must be
   updated to match, or `UnsatisfiedLinkError`/silent no-hook at runtime
   results. This also means Path A's injected `GuardApplication` and Path
   B's `App.kt` cannot coexist in the same app targeting the same JNI
   symbol from two different compiled classes with different behavior --
   pick one path per app.

9. **`axml_patch.py` (Path A) only changes an existing attribute's value,
   it does not add a missing one.** If `<application>` has no
   `android:name` attribute at all, `pack_existing_apk.sh` fails at the
   manifest-patch step. Handled correctly: the more common case where
   `android:name` IS present, either as the Android default
   (`android.app.Application`, silently proceeds) or as a real custom
   class (refuses unless `--force-application-overwrite`, since injection
   *replaces* that class's logic rather than merging into it -- there's no
   general way to merge into unknown existing bytecode without much more
   invasive smali-level patching, which is explicitly out of scope for v1).
   A non-destructive injection that *coexists* with an existing custom
   Application is possible and worth building for the common case -- either
   (a) generate `GuardApplication` to `extend` the app's existing class and
   `super`-chain it (reuses the attribute-only manifest patch), or (b)
   inject an auto-init `<ContentProvider>` whose `onCreate` runs the hook
   (touches nothing about the Application, but needs the manifest to already
   carry the provider attribute strings + resource-map entries -- which
   provider-heavy apps, e.g. anything with Firebase/androidx-startup,
   already do). Neither is implemented yet; see limitation #10 for when even
   this is moot.

10. **Apps that verify their own integrity/signature at runtime (RASP) are
    out of scope -- packing them cannot work, by construction.** Packing
    *requires* re-signing (there is no original signing key in the no-source
    path) and modifies bytes (encrypts `libapp.so`, adds a dex, patches the
    manifest). An app that checks its own APK signature or a bundle hash at
    startup will detect all of that and refuse to run -- no amount of
    cleaner injection or *smaller* encryption scope helps, because the
    signature check fires regardless of whether anything was encrypted, and
    a bundle-hash check fires on any single changed byte. This is not a bug
    in the packer; it is the target app deliberately defending against
    exactly this repackaging. **How to detect one before wasting a device
    cycle:** look in the APK for a commercial RASP/app-shielding SDK. The
    real example that motivated this note is **V-Key V-OS** (a hardened
    "Authenticator" app): tells include native libs `libvosWrapperEx.so`,
    `libloadTA.so`/`libtaInterface.so` (Trusted-App loader), `libchecks.so`,
    `libZeroCore.so`/`libzfcore.so`; assets `voscodesign.vky` (a V-OS
    code-signing blob) and `vkeylicensepack` (contains a `BndlHsh` =
    bundle-hash field); and a `<provider android:name=...VGStartUpDetector>`
    plus `zygotePreloadName=vkey.android.vos.VosZygotePreload` in the
    manifest. Other vendors (Promon SHIELD, Guardsquare/DexGuard RASP,
    Appdome, ziMperium) leave similar fingerprints. **One-command
    confirmation:** re-sign the *untouched* APK with your test key
    (`apksigner sign`, no repack/encrypt/inject) and launch it -- if it
    crashes/exits, the app rejects any signature but its own and packing is
    impossible for it; if it runs, RASP is not the blocker and the custom
    Application injection (limitation #9) is the real, solvable issue.
    Separately worth noting: an app already shipping full RASP has *stronger*
    protection than this packer adds (we defend against static snapshot
    analysis at rest; RASP defends the running process too), so packing it
    would add little even if it worked -- it is the wrong target class, not
    a failure of the tool.

    **Two distinct failure modes -- don't conflate them (learned from the
    V-OS example).** RASP apps can break packing at two different points,
    and the *observed* VSA crash was the first, not the (assumed) second:
    (i) *Runtime class loading skipped.* V-OS ships the app's own classes
    ENCRYPTED in `assets/` (e.g. `crypto_ta.bin`, content-addressed
    `assets/data/internal/<sha256>` blobs) and installs them into the
    classloader from the real `Application` class's early lifecycle
    (`attachBaseContext`). VSA's `Application` (`ertil.skauf`, present in
    `classes.dex`) does this; the protected classes -- including the
    manifest-declared `<provider>` `com.vkey.android.vguard.VGStartUpDetector`
    -- are in NONE of the dex files. Overwriting `android:name` with
    `--force-application-overwrite` discards that bootstrap, so V-OS never
    installs its classes and the app dies with `ClassNotFoundException:
    ...VGStartUpDetector` at provider creation -- *before* any integrity
    check. A non-destructive injection (subclass the existing Application and
    `super`-chain it, limitation #9) would preserve the bootstrap and get
    past this. (ii) *Integrity/signature rejection.* Only reached if (i) is
    solved: V-OS then verifies the bundle hash / code-sign and would detect
    the re-sign + encrypted `libapp.so` + added dex. The one-command re-sign
    test isolates (ii) from (i): a re-signed-but-otherwise-untouched APK runs
    `ertil.skauf` normally, so if it still crashes the blocker is integrity;
    if it runs, only the injection point (i) needs fixing.

    **RESOLVED empirically for VSA -- it is NOT integrity-blocked.** Both
    tests were run on the real device: (1) the re-signed-but-untouched APK
    launched fine (V-OS accepts a foreign signature), and (2) an
    `--encrypt-only` build (encrypted `libapp.so`, re-signed, **no**
    injection of any kind) booted V-OS fully -- it loaded `libfjvmoyxzvf.so`,
    decompressed and installed its protected `Anonymous-DexFile` (the vkey
    classes), started `DevelopMainActivity`, and created the `FlutterEngine`
    -- and only *then* `SIGSEGV`'d inside `libflutter.so` at
    `FlutterJNI.performNativeAttach` on a wild pointer, with **no** tamper/
    integrity/signature message anywhere in `logcat`. That is exactly the
    "garbage instructions" crash of an un-decrypted snapshot, at the same
    frame as the testapp's key-mismatch crash. Conclusion: **V-OS's bundle
    hash / code-sign does NOT cover `libapp.so`** (nor does it reject
    re-signing), so the encryption itself is viable on this app. The only
    real blocker was failure mode (i) -- the destructive Application
    overwrite. The fix is any *non-destructive* injection that leaves
    `ertil.skauf` intact; the minimal one (no Java at all) is to add
    `libguard.so` as a `DT_NEEDED` of `libflutter.so` and bake the key into
    it (limitation #9's option, generalized). So a V-OS-class RASP app is
    NOT automatically out of scope -- it depends entirely on whether the
    RASP's integrity coverage includes `libapp.so`, which the two tests
    above determine cheaply.

## Build vs. buy

Per `Handover.md` §13, "build" was confirmed and implemented above (custom
`libguard.so`, no license cost, full control). The maintenance cost this
carries: the `dlopen`/`android_dlopen_ext` GOT-hook technique and the
`_kDart*` symbol names are Flutter-engine-internal and NOT a stable public
API -- a future Flutter engine upgrade could rename or restructure them,
silently breaking this packer (it would fail the "app installs and all
major flows work" acceptance criterion, hopefully loudly via the
fail-closed `abort()` paths rather than silently). Re-run the ELF
verification step (re-extract `lib/<abi>/libapp.so` from a fresh build and
confirm the four `_kDart*` symbols and `dlopen`'s `JUMP_SLOT` relocation
still look as documented in `Handover.md` §3) after every Flutter SDK
upgrade, not just after Android NDK/SDK upgrades. If that maintenance
burden turns out to be higher than expected in practice, re-evaluate
against a commercial packer at that point -- no commercial alternatives
were evaluated in this pass.
